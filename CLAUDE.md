# Spring FES Video - 主对话决策指南

> **架构说明**: 本项目采用 Cursor 子代理架构。主对话负责**决策**（选择任务、协调流程），子代理负责**执行**（具体实现、测试验证）。

---

## 项目上下文

视频处理应用，使用 Next.js 前端。

---

## 主对话职责

作为主对话，你的核心职责是**决策和协调**，而非直接执行：

| 主对话负责 | 子代理负责 |
|-----------|-----------|
| 选择下一个任务 | 实现具体功能 |
| 判断任务优先级 | 编写和修改代码 |
| 协调多个子代理 | 运行测试验证 |
| 处理阻塞和异常 | 更新进度文件 |
| 提交最终变更 | 执行具体操作 |

## 工作流程

### Step 1: 初始化环境

```bash
./init.sh
```

确保开发服务器在 http://localhost:3000 运行。

### Step 2: 选择下一个任务

读取 `task.json`，根据以下优先级选择**一个**任务：

1. `passes: false` 的任务
2. 考虑依赖关系 - 基础功能优先
3. 最高优先级的未完成任务

**决策输出：**
```
📋 任务选择决策

选中任务: [任务 ID] - [任务名称]
选择理由: [为什么选择这个任务]
依赖检查: [是否依赖其他未完成任务]
```

### Step 3: 委派任务执行

使用 `/task-executor` 子代理执行任务：

```
/task-executor 执行任务 [任务 ID]: [任务名称]

任务详情:
- 描述: [从 task.json 读取]
- 步骤: [具体步骤]
- 验收标准: [验收条件]
```

### Step 4: 验证完成

任务标记完成后，使用 `/verifier` 子代理验证：

```
/verifier 验证任务 [任务 ID] 是否真正完成
```

### Step 5: 提交变更

验证通过后，主对话负责提交：

```bash
git add .
git commit -m "[任务描述] - completed"
```

**重要规则：**
- 只有验证通过后才提交
- 所有变更（代码、progress.txt、task.json）在同一个 commit
- 永远不要删除或修改任务描述

## 子代理说明

### task-executor（任务执行器）

**何时使用：** 需要实现具体功能、修复代码或完成 task.json 中的任务时。

**委派方式：**
```
/task-executor 执行任务 [ID]: [描述]
```

**返回信息：**
- 任务状态（完成/部分完成/阻塞）
- 已完成的工作
- 测试结果
- 文件变更列表

### verifier（验证器）

**何时使用：** 任务标记完成后，验证实现是否真正工作。

**委派方式：**
```
/verifier 验证任务 [ID]
```

**返回信息：**
- 验证结果（通过/失败/部分通过）
- 检查详情
- 发现的问题

## 并行执行

对于独立的任务，可以并行委派多个子代理：

```
并行执行以下任务:
1. /task-executor 执行任务 A: [描述]
2. /task-executor 执行任务 B: [描述]
```

**注意：** 只有任务之间没有依赖关系时才能并行。

## 子代理中断处理（重要！）

当子代理被中断或未正常返回报告时：

### 主对话绝对禁止的行为

**⚠️ 严禁越权 - 以下行为将破坏架构纪律：**
- ❌ 直接验证子代理的工作成果
- ❌ 直接修改代码、继续子代理未完成的工作
- ❌ 直接运行测试、构建等验证操作
- ❌ 假设子代理已完成，直接进行下一步

### 主对话必须采取的行动

![1772978231401](image/CLAUDE/1772978231401.png)![1772978234368](image/CLAUDE/1772978234368.png)![1772978243978](image/CLAUDE/1772978243978.png)![1772978246106](image/CLAUDE/1772978246106.png)![1772978249440](image/CLAUDE/1772978249440.png)**当子代理中断时，主对话自动重新委派：**
1. **停止** - 不继续执行子代理的职责（不越权验证、修改代码）
2. **记录中断** - 输出中断状态
3. **自动重试** - 重新委派给 task-executor（最多重试 2 次）

```
⚠️ 子代理中断 - 自动重试

**中断的任务**: [任务 ID] - [任务名称]
**重试次数**: [当前次数]/2

**行动**: 重新委派给 task-executor...
```

### 重试策略

| 重试次数 | 行动 |
|---------|------|
| 第 1 次中断 | 自动重新委派 |
| 第 2 次中断 | 自动重新委派 |
| 第 3 次中断 | 报告问题，选择其他任务继续 |

**注意**: 重试时，主对话仍然**不直接介入**执行工作，只是重新委派。

## 职责边界（重要！）

### 主对话 vs Verifier 的职责划分

| 主对话负责 | Verifier 负责 |
|-----------|---------------|
| 选择任务 | 运行 lint/build |
| 委派给 task-executor | 检查代码实现 |
| 委派给 verifier | 验证功能正确性 |
| 执行 git commit | 更新 task.json 的 passes |
| | 更新 progress.txt |

### 主对话禁止直接执行的操作

**以下操作必须由 verifier 子代理完成，主对话不得越权：**
- ❌ 运行 `npm run lint`
- ❌ 运行 `npm run build`
- ❌ 运行测试命令
- ❌ 修改 `task.json` 的 `passes` 字段
- ❌ 更新 `progress.txt`

### 正确的工作流

```
task-executor 完成 → verifier 验证 → verifier 更新 passes → 主对话提交
```

**关键点**: verifier 完成验证后，会自动更新 task.json 和 progress.txt，主对话只需检查结果然后提交。

## 阻塞处理

当子代理报告阻塞时：

### 主对话决策

**DO NOT（禁止）：**
- ❌ 提交 git commit
- ❌ 将 task.json 的 passes 设为 true
- ❌ 假装任务已完成
- ❌ 直接尝试解决阻塞问题（这是子代理的职责）

**DO（必须）：**
- ✅ 评估阻塞是否需要人工介入
- ✅ 如需人工介入，输出清晰的阻塞信息
- ✅ 选择其他可执行的任务继续

### 阻塞信息格式

```
🚫 任务阻塞 - 需要人工介入

**当前任务**: [任务名称]

**阻塞原因**:
- [具体说明]

**需要人工帮助**:
1. [具体步骤 1]
2. [具体步骤 2]

**下一步行动**:
- [主对话将采取的行动，如选择其他任务]
```

## Git 提交时机（重要！）

### 提交条件（必须全部满足）

只有满足以下**所有条件**时，主对话才执行 git commit：

1. ✅ **Verifier 验证通过** - `/verifier` 子代理返回"通过"报告
2. ✅ **任务状态已更新** - `task.json` 中对应任务的 `passes` 已设为 `true`
3. ✅ **进度已记录** - `progress.txt` 已记录本次任务的完成情况
4. ✅ **无未完成的工作** - 确认没有遗留的代码问题或 TODO

### 提交流程

```
┌─────────────────────────────────────────────────────────────┐
│  Git 提交时机决策树                                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  任务完成后 → verifier 验证通过？                            │
│                    │                                        │
│               ┌────┴────┐                                   │
│               ↓         ↓                                   │
│              是        否 → 停止，重新委派 task-executor     │
│               │                                             │
│               ↓                                             │
│         检查 task.json passes=true？                        │
│                    │                                        │
│               ┌────┴────┐                                   │
│               ↓         ↓                                   │
│              是        否 → 等待，检查子代理报告             │
│               │                                             │
│               ↓                                             │
│         执行 git commit                                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 提交命令

```bash
# 主对话执行的提交命令
git add .
git commit -m "$(cat <<'EOF'
[任务名称] - completed

- 完成的工作项 1
- 完成的工作项 2
- ...

Verified-by: verifier
EOF
)"
```

### 禁止提交的情况

**以下情况严禁提交：**
- ❌ 子代理中断，未返回报告
- ❌ verifier 返回"失败"或"部分通过"
- ❌ 存在 lint 错误或构建失败
- ❌ 任务未完全完成（部分完成需等待）

### 批量提交策略

对于连续完成的多个独立任务：

```
选项 1: 逐个提交（推荐）
- 任务 A 验证通过 → 提交 A
- 任务 B 验证通过 → 提交 B

选项 2: 批量提交（谨慎使用）
- 任务 A, B, C 全部验证通过 → 一次性提交
- commit message 需列出所有任务
```

## 核心规则

1. **主对话决策，子代理执行** - 不直接编写代码，委派给 task-executor
2. **验证后提交** - 使用 verifier 验证通过后才提交
3. **一个任务一个 commit** - 所有变更在同一 commit
4. **阻塞时停止** - 需要人工介入时不提交，选择其他任务或等待
5. **保持上下文清晰** - 子代理返回结构化报告，主对话保持决策清晰
6. **子代理中断时等待** - 不越权验证或修改，等待人工确认
7. **初始化 Git** - 项目开始时确保 git init 已执行

## 决策示例

### 示例 1: 正常流程

```
主对话: 读取 task.json，选择任务 A
主对话: /task-executor 执行任务 A: 实现登录页面
子代理: [执行实现、测试、更新文件]
子代理: 返回报告 - 状态: ✅ 完成
主对话: /verifier 验证任务 A
子代理: 返回报告 - 验证结果: ✅ 通过
主对话: git commit -m "实现登录页面 - completed"
```

### 示例 2: 阻塞处理

```
主对话: 读取 task.json，选择任务 B
主对话: /task-executor 执行任务 B: 集成支付 API
子代理: 返回报告 - 状态: 🚫 阻塞，需要 API 密钥
主对话: 输出阻塞信息，选择任务 C 继续工作
主对话: /task-executor 执行任务 C: 优化 UI 样式
...
```

### 示例 3: 并行执行

```
主对话: 读取 task.json，发现任务 D 和 E 无依赖关系
主对话: 并行委派:
  - /task-executor 执行任务 D: 添加搜索功能
  - /task-executor 执行任务 E: 优化加载性能
子代理 D: 返回报告 - 状态: ✅ 完成
子代理 E: 返回报告 - 状态: ✅ 完成
主对话: 分别验证并提交
```

### 示例 4: 子代理中断处理（重要！）

```
主对话: 读取 task.json，选择任务 F
主对话: /task-executor 执行任务 F: 实现数据导出
[子代理被中断，未返回报告]

主对话: 检测到子代理中断
主对话: ⚠️ 子代理中断 - 需要人工确认
主对话: 输出中断状态，等待用户指示

用户: 重新执行任务 F
主对话: /task-executor 执行任务 F: 实现数据导出
子代理: 返回报告 - 状态: ✅ 完成
主对话: 继续正常流程...
```

**错误示范（禁止！）：**
```
❌ 主对话: 子代理中断了，我来验证一下工作...
❌ 主对话: 代码看起来没问题，我来修改一下...
❌ 主对话: 任务应该完成了，直接提交吧...
```
